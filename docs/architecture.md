# 技術仕様書 (Architecture Design Document)

> 作成日: 2026-06-08
> 入力: `docs/product-requirements.md` / `docs/functional-design.md`
> 方針: サーバーを持たないブラウザ完結型SPA。静的アセットをGitHub Pagesへ配信し、データは端末内localStorageに保存する。

## アーキテクチャパターン

### レイヤードアーキテクチャ

機能設計書の3層構成に対応する。依存は常に上から下への一方向。

```
┌──────────────────────────────────────────────┐
│  UIレイヤー                                    │
│  ShopListView / RouletteView                  │ ← DOM描画・操作受付・演出
├──────────────────────────────────────────────┤
│  サービス / エンジンレイヤー                     │
│  ShopService / RouletteEngine                 │ ← ビジネスロジック・抽選計算
├──────────────────────────────────────────────┤
│  データレイヤー                                 │
│  ShopRepository                               │ ← localStorage永続化
└──────────────────────────────────────────────┘
                   ↓
              localStorage
```

依存方向のルール:
```
UI → Service/Engine → Data   (OK)
UI ← Service                 (NG: ServiceはUIを知らない)
UI → Data                    (NG: Viewは直接localStorageを触らない)
```

#### UIレイヤー（ShopListView / RouletteView）
- **責務**: DOM描画、ユーザー操作の受付と入力バリデーションのUI反映、状態のCSSクラス制御、当選演出。
- **許可される操作**: Service/Engineの呼び出し、canvas-confettiの呼び出し。
- **禁止される操作**: localStorageへの直接アクセス、ビジネスロジックの実装。

#### サービス / エンジンレイヤー（ShopService / RouletteEngine）
- **責務**: お店のCRUD・対象管理・ソート（Service）、回転/減速/当選判定の計算（Engine）。
- **許可される操作**: Repositoryの呼び出し（Service）。`requestAnimationFrame`等のブラウザAPI利用（Engine）。
- **禁止される操作**: DOM操作、UIへの依存。

#### データレイヤー（ShopRepository）
- **責務**: localStorageの読み書き、JSONシリアライズ、破損/容量超過のフォールバック。
- **許可される操作**: `localStorage` API。
- **禁止される操作**: ビジネスロジックの実装。

> **RouletteEngineをServiceと並列に置く理由**: 抽選・アニメーション計算は永続化を伴わない純粋ロジックで、`ShopService`（CRUD）とは関心が異なる。並列の独立コンポーネントにすることで、Engine単体でテスト・調整（イージング微調整）しやすくする。

## データ永続化戦略

### ストレージ方式

| データ種別 | ストレージ | フォーマット | 理由 |
|-----------|----------|-------------|------|
| お店データ（Shop[]） | localStorage（キー: `gohan-spin:shops`） | JSON文字列（`{version, shops}`） | サーバーレス・アカウント不要要件。数十件規模に十分軽量。同期APIで実装がシンプル |

### スキーマバージョニングとマイグレーション

```typescript
interface ShopStoreSchema {
  version: number;   // 現在: 1。スキーマ変更時にインクリメント
  shops: Shop[];
}
```

- 読み込み時に`version`を確認し、将来の項目追加（Post-MVPの`Shop.url`等）時はマイグレーション関数で旧→新へ変換。
- 値は文字列のため`JSON.stringify`/`JSON.parse`で出し入れ。

### バックアップ / データ保護戦略

- **自動バックアップ**: MVPでは行わない（サーバーレス・単一端末前提）。
- **破損耐性**: 読み込み時に`try/catch`し、JSON破損・`null`時は**空データで初期化して継続**（アプリをクラッシュさせない）。これがlocalStorage構成における実質的な「フォールバック戦略」。
- **将来検討**: お店データのJSONエクスポート/インポート機能を設ければ、手動バックアップと端末移行が可能（Post-MVP拡張候補）。

## パフォーマンス要件

### レスポンスタイム

| 操作 | 目標時間 | 測定環境 / 方法 |
|------|---------|----------------|
| 初期表示（FCP） | 1.5秒以内 | デスクトップChrome最新版・GitHub Pages配信。Lighthouse/Performanceで計測 |
| 一覧再描画 | 数十件で体感遅延なし（目安100ms以内） | `performance.now()`で描画前後差分を計測 |
| ルーレット回転 | 60fps（1フレーム16.7ms以内）維持 | DevTools Performanceでフレーム落ち確認。iPhone実機でも体感の滑らかさを確認 |

### リソース使用量

| リソース | 目安 | 理由 |
|---------|------|------|
| localStorage | 数十KB程度（上限の概ね5MBに対し余裕） | お店数十件×小さなJSON。容量逼迫しない |
| メモリ | 軽量（DOM要素数十・canvas紙吹雪一時生成） | 単一画面・小規模データ |

## セキュリティアーキテクチャ

### データ保護

- **暗号化**: なし（機微情報を扱わないため不要）。
- **機密情報管理**: APIキー・パスワード等を**持たない/ハードコードしない**。フロントのバンドルJSは誰でも閲覧可能なため、シークレットを置かない。
- **localStorage方針**: オリジン内JSから自由に読めるため、機微情報は保存しない（お店名・アイコンのみ＝機微でない）。
- **配信**: GitHub Pagesは既定でHTTPS。

### 入力検証

- **バリデーション**: 店名はtrim後1〜50文字必須、`iconKey`は許可値のみ（`ShopService`で検証）。
- **サニタイゼーション / XSS対策（最重要）**: ユーザー入力（店名）をDOMへ反映する際は必ず`textContent`を使い、`innerHTML`への生埋め込みを禁止。店名に`<script>`等が含まれても無害化される。
- **エラーハンドリング**: 失敗理由をユーザーに分かる日本語で表示し、内部情報（スタックトレース等）は露出しない。

## スケーラビリティ設計

### データ増加への対応

- **想定データ量**: 数十件のお店（外食先の現実的な選択肢数）。
- **localStorageの制約**: 1オリジン概ね5MB上限、同期APIでメインスレッドをブロックしうる。本件の規模では問題にならない。
- **将来データが増える場合の対策**: (1)IndexedDBへの移行、(2)一覧の遅延描画/仮想スクロール、(3)保存スキーマの`version`を活かしたマイグレーション。いずれもRepositoryの差し替えで吸収できる（レイヤー分離の効果）。

### 機能拡張性（Post-MVP）

- **外部リンク(P1)**: `Shop.url?`追加＋`version`インクリメント。データレイヤーのマイグレーションで吸収。
- **SVGアイコン(P1)**: `IconKey`抽象化により、`IconDef`の解決先を絵文字→SVGへ差し替えるだけで対応。
- **効果音(P1)**: Start操作起点で`AudioContext`初期化し、iOS/Chromeの自動再生制限を回避。
- **設定のカスタマイズ**: 将来、減速時間やリーチ強度を設定として外出しできる構造を意識（Engineのパラメータ化）。

## テスト戦略

### ユニットテスト
- **フレームワーク**: Vitest
- **対象**: `ShopService`（CRUD・バリデーション境界・ソート）、`RouletteEngine`（`getWinner`の角度判定・`build`のシャッフル網羅・N=1）、イージング関数の単調性。
- **カバレッジ目標**: ロジック層（Service/Engine/Repository）で80%以上を目安。

### 統合テスト
- **方法**: Vitest + jsdom
- **対象**: `ShopService`＋`ShopRepository`の永続化ラウンドトリップ（保存→再読込で一致）、破損JSON投入時に例外を投げず空配列で継続すること。

### E2Eテスト
- **ツール**: MVPは手動（チェックリスト）。将来Playwright導入を検討。
- **シナリオ**: 登録→一覧→対象切替→Start→Stop→当選演出の一連フロー。iPhone Chrome（実機/レスポンシブエミュレート）でホイール・ボタンが画面内に収まり操作できること。

## 技術的制約

### 環境要件
- **対象ブラウザ（MVP）**: デスクトップChrome最新版、iPhoneのChromeアプリ最新版（iOS仕様によりレンダリングエンジンはWebKit）。
- **開発環境**: devcontainer、Node.js v24系、npm 11系。
- **外部依存**: 実行時はcanvas-confettiのみ。ネットワーク通信なし。

### パフォーマンス制約
- ルーレット回転中は重い同期処理（巨大JSON化・大量DOM操作）をフレーム内に入れない（`requestAnimationFrame`で分散）。
- localStorageは同期APIのため、保存は操作イベント単位の小さなJSONに留める。

### セキュリティ制約
- フロントエンドにシークレットを置かない。
- ユーザー入力は`textContent`経由でのみDOM反映（XSS防止）。
- 利用ライブラリはMIT互換ライセンスに限定。

## 依存関係管理

| ライブラリ | 用途 | バージョン管理方針 |
|-----------|------|-------------------|
| canvas-confetti | 紙吹雪演出（実行時） | `^`（マイナーまで許可） |
| vite | ビルド/開発サーバー | `^`（マイナーまで許可） |
| typescript | 型システム | `~`（パッチのみ自動）※破壊的変更を避ける |
| vitest / @vitest/* | テスト | `^` |
| eslint / typescript-eslint / prettier | 静的解析・整形 | `^` |
| husky / lint-staged | コミット前フック | `^` |

**方針**:
- 実行時依存（canvas-confetti）は安定志向で`^`。
- TypeScriptは言語仕様の破壊的変更を避けるため`~`（パッチのみ自動更新）。
- 開発ツールは`^`で運用し、メジャー更新時のみ手動で検証して上げる。
