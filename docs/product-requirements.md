# プロダクト要求定義書 (Product Requirements Document)

> 作成日: 2026-06-08
> 入力: `docs/ideas/initial-requirements.md`（壁打ち成果物）

## プロダクト概要

### 名称
**gohan-spin** - 外食のお店をルーレットで決める、家族で盛り上がるブラウザWebアプリ

### プロダクトコンセプト
- **インストール不要**: ブラウザでURLを開くだけ。アカウント登録もサーバーも不要（GitHub Pagesで配信）。
- **演出で"体験"にする**: 単に決めるだけでなく、リーチ演出・点滅ズーム・紙吹雪で「家族みんなでドキドキする時間」を作る。
- **自分のお店リストで完結**: よく行くお店を登録しておき、その日の気分でルーレット対象を選んで回す。データは端末内（localStorage）に保存。
- **PC・スマホ両対応**: デスクトップChromeとiPhoneのChromeアプリの両方で快適に動作する、レスポンシブなUIを提供する。

### プロダクトビジョン
外食のたびに繰り返される「どこ行こう？」という迷いの時間を、家族みんなで画面を囲んで盛り上がる楽しいイベントに変える。
インストールやログインの手間なく、ブラウザを開いた瞬間に使えるサーバーレスなWebアプリとして提供する。
本プロダクトの北極星は「決めるツール」ではなく「みんなでドキドキする体験」であり、演出の楽しさこそが他の単純なルーレットツールとの差別化である。

### 目的
- 外食先を決める「迷い時間」を、家族で楽しめるエンタメ体験に変える。
- インストール・アカウント登録ゼロで、思い立った瞬間に使えるようにする。
- よく行くお店を登録・管理し、その日の気分で対象を絞ってルーレットを回せるようにする。
- 将来的な機能追加（外部リンク・SVGアイコン・効果音・他ブラウザ対応）を見据えた拡張可能な設計の土台を作る。

## ターゲットユーザー

### プライマリーペルソナ: ユッカ（外食好きの本人＋家族）
- 近所に外食の選択肢（ハンバーガー、ラーメン、寿司など）がたくさんあり、毎回どこに行くか迷う。
- 家族みんなで画面を見ながら決めたい（複数人が同じ画面を囲む利用シーン）。
- 利用環境はデスクトップChromeと、外出先・お店選びの場面ではiPhoneのChromeアプリの両方を想定する。
- ITの専門知識は必須ではないが、PC・スマホのブラウザは日常的に使う。
- アプリのインストールやアカウント登録は面倒に感じる。シンプルで見た目が楽しいものを好む。
- 典型的な利用シーン: 週末の夕食前、家族で「今日どこ行く？」となったときに画面を開き、対象店を選んでルーレットを回し、当選演出で盛り上がってからお店に向かう。

## 成功指標(KPI)

> 本プロダクトはサーバーレス（外部送信なし）のため、KPIは外部計測ツールに依存しない定性的・自己観測ベースの指標を中心に置く。プライバシー保護を優先し、利用データの外部収集は行わない。

### プライマリーKPI
- **コア体験の完走率**: 手動テストでStart→Stop→当選演出の一連フローを10回実施し、途中でアプリが停止・エラー表示になったケースが1回以下（10回中9回以上完走）。初回リリース時に開発者が実施。測定環境: デスクトップChrome最新版 / iPhone Chrome実機。
- **演出満足度**: リリース後1ヶ月以内に、実際に家族（2名以上）とルーレットを使用し、「また使いたい」「盛り上がった」という感想を1名以上から口頭で得られること。開発者が観察・記録する。

### セカンダリーKPI
- **登録のしやすさ**: 新規ユーザーが説明なしで最初のお店を1分以内に登録できる（ユーザビリティ観察）。
- **再利用（定性）**: 開発者自身が外食のたびに使いたいと感じ、週1回以上の利用が定着する（リリース後3ヶ月時点での主観記録。サーバーレス・プライバシー優先のため外部計測は行わない）。

## 機能要件

### コア機能(MVP)

#### お店の登録

**ユーザーストーリー**:
外食先を管理したいユーザーとして、よく行くお店を後で選べるように、店名とアイコンを指定してお店を登録したい。

**受け入れ条件**:
- [x] 店名（テキスト）を入力して登録できる。
- [x] 主要カテゴリからアイコンを選んで設定できる（MVPは絵文字で実装。暫定セット: 🍔ハンバーガー / 🍜ラーメン / 🍕ピザ / 🍣寿司 / 🍛カレー / ☕カフェ / その他。MVPの最終アイコンセットは機能設計書で定義し、実装前に確定する）。
- [x] アイコンは内部的にキー文字列で保持し、将来SVGアイコンセットへ差し替え可能な抽象化になっている。
- [x] 登録直後のお店は、ルーレット対象チェックがデフォルトでONになる。
- [x] 店名未入力など必須項目が欠ける場合は登録できず、理由が画面に表示される。
- [ ] お店は最大100件まで登録できる（数十件規模の利用を想定。100件時点でも一覧表示・ルーレット描画に体感遅延がない）。<!-- 登録上限100件は実装・テスト済。括弧内「ルーレット描画の体感遅延なし」はルーレット機能の実装後に再検証 -->

**優先度**: P0(必須)

#### お店の一覧表示・管理（編集・削除・対象切替）

**ユーザーストーリー**:
登録したお店を扱うユーザーとして、内容を見直したり対象を切り替えたりできるように、一覧で編集・削除・チェック操作をしたい。

**受け入れ条件**:
- [x] 登録済みのお店を一覧表示する（デフォルトでカテゴリごとにソート）。
- [x] ページングなしで全件を画面上に表示する。
- [x] 各お店にルーレット対象チェックボックスがあり、ON/OFFを切り替えられる。
- [x] チェックOFFのお店はルーレットの対象から除外される。<!-- ShopService.listEnabled() とルーレット描画（RouletteEngine.build への供給）の両方で除外を実装済み -->
- [x] 各お店を編集（店名・アイコン・対象）できる。
- [x] 各お店を削除できる（削除後は一覧とルーレット対象から消える）。
- [x] 一覧・チェック状態・編集・削除の結果はリロード後も保持される（localStorage永続化）。

**優先度**: P0(必須)

#### ルーレット（コア体験）

**ユーザーストーリー**:
家族で外食先を決めたいユーザーとして、みんなでドキドキしながら1店に決められるように、円形ホイール型のルーレットを回したい。

**受け入れ条件**:
- [x] ルーレットは円形ホイール（運命の輪）型で表示され、上部の針が指した店が当選となる。
- [x] ルーレットにはチェックONのお店だけが表示される。
- [x] ルーレット上の並び順は一覧（カテゴリ順）とは独立し、ランダムに配置される（Fisher-Yatesシャッフル）。
- [x] Startボタンで回転を開始できる。
- [x] Stopボタンを押すとゆっくり減速しながら停止する。
- [x] 停止して決まったお店を点滅＋ズームで強調表示する。
- [x] 当選表示はアイコンと店名をセットで画面中央に大きく表示する（同一アイコンの店が複数あっても店名で一意に特定できる）。
- [x] 対象店が0件のときはStartできず、対象を選ぶよう促すメッセージを表示する。
- [x] 当選演出の後、再度ルーレットを回せる（「もう一度」ボタンで初期状態に戻る）。
- [ ] デスクトップ（横幅広）でもiPhone（縦長・狭幅）でも、ホイール全体と針・Start/Stopボタンが画面内に収まり、操作できる。<!-- レスポンシブCSS（880px境界の2カラム⇔縦積み・70vminのホイール）は実装済み。実機（iPhone Chrome）での収まり確認が未実施のため未チェック -->

**優先度**: P0(必須)

#### 当選演出（差別化の核）

**ユーザーストーリー**:
家族で盛り上がりたいユーザーとして、決定の瞬間を最大限に楽しめるように、リーチ・点滅ズーム・紙吹雪の演出が欲しい。

**受け入れ条件**:
- [x] 減速の最終局面で回転をぐっとスロー＆長くするリーチ演出を行い、緊張感を高める（easeOutQuintイージング・減速4〜6秒）。
- [x] 当選店を点滅させ、アイコン＋店名を画面中央に大きく表示する。
- [x] 当選時にお祝いの紙吹雪を表示する（`canvas-confetti`を使用・MITライセンス）。
- [ ] アニメーションがデスクトップChromeで滑らかに動作する（カクつきが体感されない）。<!-- CSS transform（GPU合成）+ requestAnimationFrame で実装済み。実機での体感確認が未実施のため未チェック -->

**優先度**: P0(必須)

### UI / 画面と操作

```
# 主要画面と操作（単一ページのSPAを想定）
- お店管理エリア:
    - 追加フォーム: 店名を入力 → カテゴリ/アイコンを選択 → [登録]
    - 一覧: カテゴリ順で全件表示。各行に [対象チェックボックス] [店名+アイコン] [編集] [削除]
- ルーレットエリア:
    - 円形ホイール（対象ON店をランダム配置）
    - [Start] / [Stop] ボタン
    - 当選表示オーバーレイ（点滅・ズーム・アイコン+店名・紙吹雪）

# レスポンシブ・レイアウト方針
- デスクトップ（横幅広）: お店管理エリアとルーレットエリアを横並びに配置し、画面を広く活用する。
- iPhone（縦長・狭幅）: 縦積みに切り替え、ルーレットを主役に大きく配置。お店管理は下方やトグル表示等で省スペース化する。
- タップ操作前提のボタンサイズ・間隔（指で押しやすい十分なヒット領域）を確保する。
```

### 将来的な機能(Post-MVP)

> これらはMVPの後、今期内に追加開発する項目。スコープ外ではなく「優先度が一段下」。設計時に拡張ポイントを確保する。

#### 各お店への外部リンク
お店のサイト等の外部URLを各お店に登録・表示できるようにする。

**優先度**: P1(重要)

#### アイコンのSVGアイコンセット化
絵文字アイコンをSVGアイコンセットへ差し替える（キー文字列抽象化を活かす）。

**優先度**: P1(重要)

#### 効果音
回転中のカチカチ音、停止時のドラムロール等。Startボタン操作起点でChromeの自動再生制限を回避する。

**優先度**: P1(重要)

#### Chrome系以外のブラウザ対応
デスクトップSafari / Firefox / Edge等での動作対応。
（※ iPhoneのChromeアプリはMVPの対象環境。iOSのブラウザはエンジンがWebKitのため、実質的にiOS Safariでの動作担保も兼ねる点に留意する。）

**優先度**: P2(できれば)

## 非機能要件

### 対象環境
- デスクトップChrome（最新版）。
- iPhoneのChromeアプリ（最新版。iOS仕様によりレンダリングエンジンはWebKit）。
- 上記2環境で、レスポンシブデザインによりレイアウトを最適化する。

### パフォーマンス
- 初期表示(First Contentful Paint): 1.5秒以内（デスクトップChrome最新版、GitHub Pages配信時）。
- ルーレットのアニメーション: 60fps（1フレーム16.7ms以内）を維持し、カクつきを体感させない（iPhone実機でも体感上の滑らかさを確保）。
- お店数十件規模の登録・一覧表示・ルーレット描画が遅延なく動作する（一覧再描画の目安: 100ms以内。詳細目標は `docs/architecture.md` を参照）。

### ユーザビリティ
- 新規ユーザーが説明なしで最初のお店を1分以内に登録できる。
- Start / Stop / 登録 / 編集 / 削除 / 対象切替の主要操作が、説明なしで直感的に分かる。
- 当選店がアイコンだけでなく店名併記で一意に伝わる。
- デスクトップ（マウス）でもiPhone（タップ）でも、主要操作が無理なく行える。タップターゲット（ボタン・チェックボックス等）は44×44px以上のヒット領域を確保する（Apple Human Interface Guidelines推奨値準拠）。

### 信頼性
- リロード・ブラウザ再起動後もお店データとチェック状態が保持される（localStorage永続化）。
- localStorageのデータが破損・不正な場合も、初期化して動作を継続し、アプリがクラッシュしない。
- ルーレット対象が0件など境界条件でも、明確なメッセージを出して安全に動作する。

### セキュリティ
- サーバー送信なし。お店データは端末内のlocalStorageにのみ保存し、外部に送信しない。
- API キー・パスワード等の機密情報を扱わない（ハードコードもしない）。
- 利用ライブラリはMITライセンス互換のものに限定し、ライセンス影響があれば再検討する。

### スケーラビリティ
- アイコンはキー文字列で抽象化し、絵文字→SVGへ差し替え可能にする。
- データモデルはお店への外部リンク等の項目追加に耐えられる構造とする（後方互換を意識）。

## スコープ外

MVP時点で明示的にスコープ外とする項目（将来検討を含む）:
- お店データの複数端末間同期・クラウド保存（サーバーレス方針のため）。
- アカウント・ログイン機能。
- 地図連携・店舗検索・口コミ等の外部サービス連携。
- ネイティブアプリ化（iOS/Android）。
- 多言語対応（まずは日本語のみ）。
